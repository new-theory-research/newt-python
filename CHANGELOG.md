# Changelog

## Unreleased

**`newt models` human output: no more raw uids, and it now says whose key you're looking at** — Mattie's 2026-07-17 friction report on the human renderer, three fixes: (1) raw uids used to appear on every line, dominating visually while serving no action a human takes browsing the catalog — they're gone from default output; the tag (fine-tunes) and display name (bases) already carried the real identity. `--ids` restores them for scripts/agents that still want the handle. (2) the listing now leads with a `Key` line naming which credential produced it — the masked key (console's own `nt_••••••••` + last-8-chars convention, apps/console/db/schema.ts) plus its source, `(environment)` or `(credentials file)`, mirroring `newt status`'s key-source resolution. The full key is never printed (test-asserted). (3) base→fine-tune grouping and the axes line are unchanged. `--json` is byte-identical to before — none of this touches the machine path. No wire change, no new network call. **Deferred, escalated rather than guessed:** an explicit "Your models" vs "Base models" section header was in scope but the `/v1/models` response has no ownership signal to hang it on — `registry_service._model_response` (nt-runway) never exposes the DB catalog's `owner` field, and the endpoint's own docstring says it "returns the composed snapshot... to any valid key (no per-model auth scoping in v0)" — every valid key sees the same fine-tunes, not just its own. Labeling them "yours" would be inventing a claim the payload doesn't support (Rule 10). Needs a wire change (expose+scope by owner) before it can ship; filed as a follow-up rather than shipped as a heuristic.

**New verb `newt upgrade` + a quiet once-a-day "you're stale" nudge** — the CLI ships fixes daily, and until now nothing told anyone they were running an old one. `newt upgrade` runs the one documented upgrade command (`uv tool upgrade newt`) when it can confirm `newt` was installed as a `uv tool`; when the install method can't be confirmed it PRINTS the command instead of running it, so it never runs a package manager against the wrong environment. `--print` always prints, never runs. Alongside it, a passive notice: after any command succeeds, `newt` may print one quiet line to **stderr** — `newt <latest> available — run 'newt upgrade'` — sourced from the console's public `GET /api/cli/version`. It is do-no-harm by construction: never blocks or slows a command (post-output, ~1s cap), fires at most once a day via a `~/.nt/update-check.json` timestamp cache, is silent on any network failure (a dead endpoint costs nothing), is stderr-only, is skipped entirely under `--json`, and is disabled by `NEWT_NO_UPDATE_CHECK=1`.

**Release convention — bump the version when you ship a CLI change.** The documented install (`uv tool install "git+ssh://…"`) tracks `main`, and the passive check compares this package's version (`importlib.metadata.version("newt")`) against the console's `latest`. That comparison is only honest if the version actually moves when we ship — a version that never bumps can never fire the notice. So bumping `pyproject.toml` `version` here **and** the console constant (`apps/console/lib/cli-version.ts` `LATEST_CLI_VERSION`) is now part of shipping a CLI release, in lockstep. This release bumps `0.0.1 → 0.0.2` to make the signal live.

**New `robot.contract` + local observation pre-flight** — the SDK already fetches each model's contract on construction (state shape, cameras, image shape, action shape and axes) but kept it private, so a starter that wanted to show "your install resolved the contract" reached into `robot._registry` by hand, and a wrong-shaped observation waited out a ~50s cold-start round-trip only to come back with a shape-mismatch the SDK could have named in microseconds. Both are now local, instant facts. `robot.contract` is a read-only view (`newt.ModelContract`) over the already-fetched registry entry — `robot.contract.state_shape`, `.cameras`, `.image_shape`, `.action_shape`, `.action_axes`, plus `.raw` for any other field the payload carries — resolving to `None` (never a fabricated default) when the model has no contract or `NT_INFERENCE_URL` is set. And `run()`/`infer()` now pre-flight the observation against that contract **before a single byte hits the wire**: a wrong state dim, an unknown camera key, or a wrong image shape raises `ContractMismatchError` instantly, client-side, naming expected-vs-got — while a valid observation ships the byte-identical frame it always did, and a partial observation the server firehose-coerces is left alone. The check runs once per call (`run()` validates the first frame only), so the streaming hot path is untouched. No wire change, no new fetch. Riders: `RegistryUnavailable` now subclasses `NewTheoryError` (so `except newt.NewTheoryError` catches a registry outage), `Robot.run()` without callbacks raises a `NewTheoryError` (was a bare `TypeError`), a `py.typed` marker ships so the SDK's type hints reach mypy/pyright/IDEs/agents, `newt.fixtures` now resolves as the documented deprecated alias (warning on use, not on `import newt`), and the state/action docstrings stop hardcoding `(14,)` — they point at `robot.contract`.

**New verb `newt run <tag>`** — the CLI hero verb: take a model tag, authenticate, load a real bundled observation, and call the model **once against prod**, then print the resolved model, the round-trip latency, and the action-chunk shape. Built on the parts that already ship — `Robot(model=tag)`'s in-constructor tag→registry resolve, the callback-free one-shot `infer(obs)`, and the self-describing `newt.snapshots`. Flags: `--snapshot <name>` (default `cup_stacking`; also `pour_coffee_beans`), `--prompt "..."` to override the snapshot's recorded prompt, and `--json` for the machine-readable mirror. **v1 is hardware-free by design** — no robot is connected and nothing moves; the human output says so plainly so a live inference is never mistaken for a robot demo. Errors render in the house shape (`newt: <problem> — <hint>`, no-key → `newt login`), and a server detail carrying a pending/dead own-model `model_status` is surfaced **verbatim**. `--embodiment` and the streaming loop against real hardware are a separate future phase.

New verb `newt promote <job-handle> --band <token>` — keep a fine-tune's checkpoint band and serve it from the terminal, the CLI twin of the console's promote button. The band is registered as a model born `pending`; the normal admission chain takes it live, and the output points at `newt models` to watch it. Every server refusal prints its plain reason verbatim — a band whose eval hasn't landed, a checkpoint whose location the pipeline hasn't reported yet, a run already promoted (with its existing model's uid) — never a generic failure. `--band` is passed verbatim; `--json` emits the route's response body on stdout. Reaching the promote route from the CLI required teaching it to accept a Bearer key beside its browser session (a portal-side change).

**Breaking behavior change:** `Robot()` credential resolution is now env-first — `NT_API_KEY` environment variable wins over `~/.nt/credentials` file. Previously the SDK read the file first, while CLI verbs were already env-first; this unifies both to the 12-factor convention. Only observable when both sources are set and disagree.

Default bootstrap URL flipped from the NT0-FP3 Modal serve URL to the always-on registry at https://nt-registry-production.up.railway.app (merge `f59a993`). First `Robot()` now resolves routing in about a second even when the GPU container is cold, and no longer raises `RegistryUnavailable` on a cold container. `NT_BOOTSTRAP_URL` and `NT_INFERENCE_URL` overrides are unchanged.
